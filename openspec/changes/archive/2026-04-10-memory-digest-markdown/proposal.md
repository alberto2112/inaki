# Proposal: memory-digest-markdown

**Status**: Proposed
**Date**: 2026-04-10
**Change ID**: memory-digest-markdown

---

## Intent

Replace the per-turn `memory.search` vector lookup with a stable, regenerated
markdown digest of the N most recent long-term memories. Two wins:

1. **Zero embedding calls per turn when neither skills-RAG nor tools-RAG is active.**
   Today `run_agent.py:79` calls `embed_query` unconditionally because
   `memory.search` needs the vector. After this change, `embed_query` runs
   only if `skills_rag_active or tools_rag_active`.
2. **Predictable recall of recent memories.** The system prompt always carries
   the last N memories (`created_at DESC`) instead of a similarity-based slice
   whose quality is noisy for short user turns.

The digest is a human-readable file the user can open and inspect.

---

## Scope

### In Scope

- Remove `memory.search` call from `core/use_cases/run_agent.py` hot path
  (both `execute` and `inspect`).
- Move `embed_query` inside a conditional: run only when
  `skills_rag_active or tools_rag_active`.
- `AgentContext` (`core/domain/value_objects/agent_context.py`): remove
  `memories: list[MemoryEntry]`, add `memory_digest: str` and render it in
  `build_system_prompt`.
- `ConsolidateMemoryUseCase`: after the existing extract+store loop, regenerate
  the digest file by calling the existing `memory.get_recent(cfg.memory.digest_size)`
  (already defined at `core/ports/outbound/memory_port.py:18` and implemented at
  `adapters/outbound/memory/sqlite_memory_repo.py:120` with the exact
  `ORDER BY created_at DESC LIMIT ?` query needed) and writing markdown via
  `Path.write_text`. Archive + clear of conversation history (lines 151-156)
  stays unchanged. No new port method is introduced.
- `ConsolidateMemoryUseCase.__init__`: add a new `memory_config: MemoryConfig`
  parameter (the use case today only receives `agent_id`, not the config) and
  update `infrastructure/container.py:66-72` to pass `cfg.memory` as a new kwarg.
- `MemoryConfig` (`infrastructure/config.py`): add
  `digest_size: int = 14` and `digest_path: str = "~/.inaki/mem/last_memories.md"`
  with `~` expansion resolved at load time.
- Document both fields in `config/global.example.yaml` under the `memory`
  section with inline comments.
- `RunAgentUseCase` reads the digest via `Path.read_text` at the top of each
  turn; if the file does not exist, it passes an empty string (graceful
  fallback on first run before any consolidate).
- Update `InspectResult` to replace `memories: list[MemoryEntry]` with
  `memory_digest: str`.
- Tests: unit test proving `embed_query` is NOT called when both RAG modes are
  inactive; unit test asserting `ConsolidateMemoryUseCase` calls `get_recent`
  with `cfg.memory.digest_size`; integration-style test for consolidate writing
  the digest file with the agreed format. The ordering and limit behavior of
  `get_recent` is already covered by existing adapter tests — no new adapter
  test is added for the method itself.

### Out of Scope

- Future LLM tool that lets the agent search old memories on-demand (this is
  exactly why `SQLiteMemoryRepository.search` is NOT deleted — it stays
  available for the future tool).
- General migration of user-owned paths (`data/`, `models/`, `skills/`,
  `config/`) to `~/.inaki/`. Only the digest path uses the home location in
  this change. Everything else stays project-relative.
- Cleanup/rotation protocol for the growing `data/inaki.db` (accepted risk;
  the user is aware).
- Refresh-on-startup of the digest or a `/refresh-digest` command. Digest is
  regenerated only at the end of `/consolidate`.
- Changing how facts are extracted, embedded, or stored in `inaki.db`.

---

## Capabilities

### New Capabilities

- `memory-digest`: markdown digest of recent long-term memories generated by
  consolidate and injected into every turn's system prompt.

### Modified Capabilities

- `run-agent`: no longer calls `memory.search` per turn; `embed_query` is now
  conditional on RAG activity.
- `consolidate-memory`: gains a final step that writes the digest file; keeps
  archive+clear of history unchanged.

---

## Approach

**Approach A — Inline minimal** (chosen). No new port for the digest itself.
`Path.write_text` lives in `ConsolidateMemoryUseCase`; `Path.read_text` lives
in `RunAgentUseCase`. No new port method is needed — the existing
`IMemoryRepository.get_recent(limit)` already returns memories ordered by
`created_at DESC` and is reused as-is to source the digest.

Rejected alternatives:
- **Approach B — Dedicated `IMemoryDigestStore` port + `MarkdownMemoryDigest`
  adapter.** Clean hex, but two call sites and one tiny responsibility do not
  justify a new port + adapter + wiring. If the format ever needs to evolve
  (JSON, YAML), the refactor is ~30 min.
- **Approach C — No file, inline formatting from `list_latest`.** Rejected
  because the user explicitly wants an inspectable markdown file on disk.

### User-data separation principle

This proposal is the **first** case of a user-data path living outside the
project tree (`~/.inaki/mem/last_memories.md`). The long-term intent is that
all user-owned data — configs, secrets, memories, user-installed tools and
skills — lives under `~/.inaki/`. That migration is incremental. This change
only moves the digest path; other paths stay project-relative. Reviewers of
future changes should treat this as an established direction, not a one-off.

### Markdown format

```
# Recuerdos sobre el usuario
<!-- Generado por /consolidate — 2026-04-10T14:30Z -->

- [2026-04-09] Prefiere español rioplatense (preferencias, idioma)
- [2026-04-08] Usa LazyVim y Tmux (tooling)
```

- Line format: `- [YYYY-MM-DD] {content} ({tag1}, {tag2})`
- Header comment carries ISO8601 UTC timestamp of generation.
- Order: `created_at DESC` (most recent first).

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Stale digest if memories are inserted outside `/consolidate` (tests, future features). | Med | Documented limitation. Digest is the consolidate artifact, not a live view. A future refresh mechanism is out of scope. |
| `InspectResult` debug API changes (`memories` → `memory_digest`). | Low | Mechanical; any caller of `inspect` is internal (CLI debug). Update in the same PR. |
| `data/inaki.db` grows unbounded because consolidate does not prune long-term memories. | Med | Accepted. Long-term memory cleanup is a separate future change. |
| `~/.inaki/mem/` parent directory does not exist on first consolidate. | Low | `Path.mkdir(parents=True, exist_ok=True)` before `write_text`. |
| `embed_query` still called when user expected zero calls, due to a RAG flag being true by default. | Low | Unit test asserts both flags false → no embedding call. |

---

## Rollback Plan

Revert the PR. The digest file on disk becomes orphaned but harmless; delete
it manually or leave it. `SQLiteMemoryRepository.search` was never removed, so
the previous per-turn behavior is restored verbatim. No schema migration to
undo.

  filesystem access) already exist in the codebase.
